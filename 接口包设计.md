下面给你们一版可直接编码的协议，名字先定成 **KBGP/1.0**（Kerberos Battle Game Protocol）。

我采用的是“**保留 Kerberos 语义，但线缆格式改成 WebSocket + JSON 文本帧**”的路线：AS 负责给 TGT，TGS 负责给 Service Ticket，GS 用 `ticket + authenticator` 做 AP 风格双向认证；票据本身不能单独证明身份，必须配合 authenticator、时间戳和防重放检查；认证成功后，客户端和游戏服务器共享会话密钥来保护后续业务消息。这和 Kerberos V5 在 RFC 4120 里的 AS/TGS/客户端-服务器交换、authenticator、防重放与 mutual authentication 的核心语义是一致的。([RFC 编辑器][1])

我也不是空想设计，而是按你们仓库当前状态收口：你们详细设计文档已经把目标定成“Kerberos + 双人对战 + WebSocket + 服务端权威裁决”，并且明确要结构化协议、固定 Tick、断线后按空输入推进；但当前可运行代码里，服务端实际上还是演示级 relay，只处理 `JOIN_ROOM / INPUT / CHAT / LEAVE_ROOM`，客户端当前真正序列化出去的输入 DTO 已固定为 `seq / moveX / jumpPressed / attackPressed / downHeld / dropPressed / aimX / aimY / clientState / clientGrounded / clientJumpCount / clientPosX / clientVelX`，而本地预测代码又已经在读 `dashPressed / attackHeld`。所以下面的协议会优先兼容你们**现在的 C# / Python 代码**，而不是只兼容论文式草案。([GitHub][2])

另外，你们文档里已经有一版统一 Envelope 和房间/输入/快照/事件草案，但你允许我推翻重做，所以我只保留它的目标，不保留旧字段；尤其是旧草案只有一个全局 `iv` 字段，这在一条消息里同时出现 `ticket`、`authenticator`、`payload` 三类独立密文时会有歧义，我下面会把它改成“**每个密文 blob 自带自己的 IV**”。([GitHub][2])

## 1. 总体方案

### 1.1 角色与连接顺序

1. Client 连接 AS，完成注册 / 登录 / 改密。
2. Client 连接 TGS，提交 TGT，申请访问目标 Game Server 的 Service Ticket。
3. Client 连接 GS，提交 Service Ticket 和 Authenticator，完成双向认证。
4. 认证后进入大厅 / 房间 / 对战业务流。
5. 断线重连时，重新连 GS，复用旧 `sessionId + Service Ticket`，但**必须生成新的 Authenticator**。

### 1.2 协议风格

* 传输：WebSocket 文本帧
* 编码：UTF-8 JSON
* 字段命名：**lowerCamelCase**
* 消息类型：**UPPER_SNAKE_CASE**
* 时间：UTC Unix 毫秒
* 二进制：Base64

我统一用 `lowerCamelCase`，是为了尽量少改你们现有 Unity DTO 和当前 Python relay 的字段风格。([GitHub][3])

### 1.3 默认时效

这是我建议你们直接采用的默认值：

* `clockSkewMs = 30000`
* `tgtLifetimeMs = 8 * 60 * 60 * 1000`
* `tgtRenewUntilMs = 24 * 60 * 60 * 1000`
* `serviceTicketLifetimeMs = 2 * 60 * 60 * 1000`
* `heartbeatIntervalMs = 10000`
* `heartbeatTimeoutMs = 30000`
* `reconnectGraceMs = 30000`

### 1.4 单端登录与立即失效

为满足“**不能多端同时登录**”和“**主动登出后当前票据/会话立即失效**”，我建议引入一个账号级版本号：

* `loginGen`：用户当前登录代数，保存在 `users` 表。
* 每次 **AS 登录成功**：`loginGen += 1`
* 每次 **修改密码成功**：`loginGen += 1`
* 每次 **LOGOUT 成功**：`loginGen += 1`

然后把 `loginGen` 写进 **TGT、Service Ticket、GS 会话记录**。
TGS / GS 每次校验票据时都检查票据里的 `loginGen` 是否等于数据库当前值；不等就立即拒绝。

你们文档里已经有 `ws_sessions(session_id, user_id, service_name, room_id, ticket_expire_at, last_seq, status)` 这张基础会话表，我建议在此基础上再加 `login_gen`、`resume_until_ms`、`match_id`、`last_input_seq`。([GitHub][2])

## 2. 统一外层 Envelope

所有消息都用这一层。

```json
{
  "ver": "KBGP/1.0",
  "msgType": "AS_REQ",
  "service": "AS",
  "requestId": "req-8f5f7f4c-1d2b-4f77-a73c-1d4f0d6a8fd6",
  "connSeq": 1,
  "timestampMs": 1776650400000,
  "clientId": "cli-34d9aa4d-7b90-4c24-89f8-8a3ef2f46301",
  "sessionId": null,
  "roomId": null,
  "status": "REQ",
  "security": {
    "enc": "RSA",
    "macAlg": "NONE",
    "keyId": "as-rsa-2026-01",
    "ticket": null,
    "authenticator": null,
    "mac": null
  },
  "payload": null,
  "payloadCipher": "Base64(...)",
  "error": null
}
```

### 2.1 外层字段定义

| 字段              | 类型          | 说明                                      |
| --------------- | ----------- | --------------------------------------- |
| `ver`           | string      | 固定 `"KBGP/1.0"`                         |
| `msgType`       | string      | 消息类型                                    |
| `service`       | string      | `AS / TGS / GS`                         |
| `requestId`     | string      | 请求唯一标识，幂等用                              |
| `connSeq`       | long        | **每条 WebSocket 连接、每个发送方自己的递增序号**，从 1 开始 |
| `timestampMs`   | long        | 发送时间                                    |
| `clientId`      | string      | 客户端安装实例 ID，建议首次启动生成 UUID 并持久化           |
| `sessionId`     | string/null | GS 业务会话 ID，`GS_AUTH_OK` 后才有             |
| `roomId`        | string/null | 房间 ID，进房后才有                             |
| `status`        | string      | `REQ / REP / PUSH / ERR`                |
| `security`      | object      | 安全相关字段                                  |
| `payload`       | object/null | 只放**明文公共元数据**                           |
| `payloadCipher` | string/null | 真正业务体的密文 Base64                         |
| `error`         | object/null | 错误对象                                    |

### 2.2 `security` 字段定义

| 字段              | 类型          | 说明                                 |
| --------------- | ----------- | ---------------------------------- |
| `enc`           | string      | `NONE / RSA / DES`                 |
| `macAlg`        | string      | `NONE / HMAC-SHA256 / DES-CBC-MAC` |
| `keyId`         | string/null | 公钥版本，如 `as-rsa-2026-01`            |
| `ticket`        | string/null | TGT 或 Service Ticket 的 Base64 blob |
| `authenticator` | string/null | Authenticator 的 Base64 blob        |
| `mac`           | string/null | 消息完整性校验                            |

### 2.3 为什么删掉全局 `iv`

你们旧草案的外层只有一个 `iv` 字段。那样一条消息如果同时带：

* `ticket`
* `authenticator`
* `payloadCipher`

就不知道这个 `iv` 到底是给谁用的。
所以我建议直接废掉全局 `iv`，改成：

> **每个 DES 密文 blob 自己把 IV 前置进去。**

## 3. 加密与完整性规则

### 3.1 DES 密文 blob 统一格式

所有 DES 加密对象都统一成这个格式：

```text
EncBlob_DES(key, obj) = Base64( IV(8 bytes) || DES-CBC-PKCS7(key, IV, UTF8(minJson(obj))) )
```

也就是说：

* `ticket` 是一个 `EncBlob_DES`
* `authenticator` 是一个 `EncBlob_DES`
* `payloadCipher` 也是一个 `EncBlob_DES`

### 3.2 RSA 密文格式

```text
EncBlob_RSA(pubKey, obj) = Base64( RSA-OAEP-SHA256(pubKey, UTF8(minJson(obj))) )
```

我建议用 **RSA-OAEP-SHA256**。
如果你们现有库实在不好配，也可以退到 PKCS#1 v1.5，但优先 OAEP。

### 3.3 密钥规则

* `K_user = PBKDF2-HMAC-SHA256(password, userSalt, pwdIter, dkLen=8)`
* `K_tgs`：AS / TGS 共享长期 DES 密钥
* `K_gs`：TGS / GS 共享长期 DES 密钥
* `K_c_tgs`：AS 签发给 Client-TGS 的会话密钥
* `K_c_gs`：TGS 签发给 Client-GS 的会话密钥

### 3.4 `mac` 规则

我默认这样做：

```text
K_mac = SHA256(sessionKey || "mac")
mac = Base64( HMAC-SHA256(K_mac, canonicalText) )
```

`canonicalText` 固定按下面顺序拼接，中间用 `\n`：

```text
ver
msgType
service
requestId
connSeq
timestampMs
clientId
sessionId
roomId
status
security.enc
security.macAlg
security.keyId
security.ticket
security.authenticator
payloadCipher
```

空值按空串处理。

### 3.5 关于 `macAlg`

你只明确要求了“**对称加密用单 DES**”，没有明确禁止哈希/MAC。
所以为了鲁棒性，我默认 `macAlg = HMAC-SHA256`。

如果老师要求“协议里只能出现 DES / RSA”，那就把：

* `HMAC-SHA256`

换成：

* `DES-CBC-MAC`

即可，**报文字段不需要改**，只改算法实现。

## 4. 票据、认证器、会话对象

### 4.1 TGT 明文结构

TGT 用 `K_tgs` 加密，Client 只能转发，不能解开。

```json
{
  "ticketId": "tgt-7e74d8f0-2a09-4d2d-b24b-5a75c3db8f30",
  "realm": "GAME.LOCAL",
  "userId": 10001,
  "username": "player_a",
  "clientId": "cli-34d9aa4d-7b90-4c24-89f8-8a3ef2f46301",
  "tgsName": "krbtgt/GAME.LOCAL",
  "kcTgs": "Base64(8-byte-DES-key)",
  "loginGen": 17,
  "issuedAtMs": 1776650400000,
  "expireAtMs": 1776679200000,
  "renewUntilMs": 1776736800000
}
```

### 4.2 Service Ticket 明文结构

Service Ticket 用 `K_gs` 加密。

```json
{
  "ticketId": "st-5302c9a1-03f8-4b37-b658-9864c70c1162",
  "realm": "GAME.LOCAL",
  "userId": 10001,
  "username": "player_a",
  "clientId": "cli-34d9aa4d-7b90-4c24-89f8-8a3ef2f46301",
  "serviceName": "game/ws@10.0.0.12:9100",
  "kcGs": "Base64(8-byte-DES-key)",
  "loginGen": 17,
  "issuedAtMs": 1776650500000,
  "expireAtMs": 1776657700000
}
```

### 4.3 TGS / GS Authenticator 明文结构

Authenticator 必须每次新生成，**绝不能重用**。Kerberos 标准明确要求 authenticator 不能重用，并要求服务端在允许的 clock skew 窗口内做 replay cache。([RFC 编辑器][1])

```json
{
  "clientId": "cli-34d9aa4d-7b90-4c24-89f8-8a3ef2f46301",
  "timestampMs": 1776650500100,
  "nonce": "auth-nonce-001",
  "requestId": "req-abc-001"
}
```

重连时多一个字段：

```json
{
  "clientId": "cli-34d9aa4d-7b90-4c24-89f8-8a3ef2f46301",
  "sessionId": "sess-9001",
  "timestampMs": 1776650600100,
  "nonce": "auth-nonce-002",
  "requestId": "req-reconnect-001"
}
```

### 4.4 GS 双向认证回包（AP_REP-like）

RFC 4120 要求 mutual authentication 通过 `KRB_AP_REP` 支持；V5 中客户端验证的是“服务端返回的时间戳是否与自己发出的 authenticator 一致”。([RFC 编辑器][1])

我给你们的 `GS_AUTH_OK` 解密明文定义为：

```json
{
  "clientTimestampMs": 1776650500100,
  "nonce": "auth-nonce-001",
  "serverTimeMs": 1776650500120,
  "sessionId": "sess-9001",
  "serviceTicketExpireAtMs": 1776657700000,
  "heartbeatIntervalMs": 10000,
  "reconnectGraceMs": 30000
}
```

客户端校验：

1. `clientTimestampMs` 是否等于自己发的 authenticator 时间戳
2. `nonce` 是否等于自己发的 nonce
3. 能否成功用 `K_c_gs` 解开

这三条都对，就说明 GS 确实持有票据里的会话密钥。

## 6.5 `GS_AUTH`

外层：

* `service = GS`
* `sessionId = null`
* `security.ticket = ServiceTicket`
* `security.authenticator = EncBlob_DES(K_c_gs, gsAuthenticator)`
* `payloadCipher` 可为空，也可放客户端构建号等信息

`gsAuthenticator`：

```json
{
  "clientId": "cli-34d9aa4d-7b90-4c24-89f8-8a3ef2f46301",
  "timestampMs": 1776650510000,
  "nonce": "auth-gs-001",
  "requestId": "req-gs-auth-001"
}
```

`GS_AUTH_OK.payloadCipher` 解密后：

```json
{
  "clientTimestampMs": 1776650510000,
  "nonce": "auth-gs-001",
  "serverTimeMs": 1776650510020,
  "sessionId": "sess-9001",
  "serviceTicketExpireAtMs": 1776657700100,
  "heartbeatIntervalMs": 10000,
  "reconnectGraceMs": 30000
}
```

认证完成后，后续所有 **GS 业务消息** 都：

* 不再带 `ticket`
* 不再带 `authenticator`
* 只带 `sessionId`
* 用 `K_c_gs` 做 `payloadCipher + mac`

这就是我对“按照 Kerberos 实现来”的落地解释：Kerberos 在 `AP_REQ / AP_REP` 之后，应用层直接使用双方共享的 session key；不是每个业务消息都重复携带 ticket。([RFC 编辑器][1])

---

## 6.6 `RECONNECT_REQ`

RFC 4120 允许客户端在票据未过期时重复使用票据；所以你要求的“断线重连复用旧 Service Ticket”是符合 Kerberos 语义的，但**每次重连必须重新生成新的 authenticator**。([RFC 编辑器][1])

外层：

* `service = GS`
* `sessionId = 旧 sessionId`
* `security.ticket = 旧 ServiceTicket`
* `security.authenticator = EncBlob_DES(K_c_gs, reconnectAuthenticator)`
* `payloadCipher = EncBlob_DES(K_c_gs, reconnectBody)`

`reconnectBody`：

```json
{
  "lastSeenServerConnSeq": 381,
  "lastProcessedSeq": 152
}
```

`RECONNECT_REP.payloadCipher` 解密后：

```json
{
  "resumeOk": true,
  "sessionId": "sess-9001",
  "roomId": "room-123",
  "matchId": "match-888",
  "phase": "PLAYING",
  "serverTimeMs": 1776650610000
}
```

如果成功，GS 立刻继续推：

1. `ROOM_STATE_PUSH`
2. `MATCH_SNAPSHOT_PUSH`

### 重连规则

* `sessionId` 必须存在且状态为 `DISCONNECTED_GRACE`
* `Service Ticket` 必须仍未过期
* `loginGen` 必须仍然匹配
* 距离断线时间不能超过 `reconnectGraceMs`
* 超过 30 秒，返回 `GS_RECONNECT_TIMEOUT`

## 7. 正常流程时序

## 7.1 注册

1. C -> AS `REGISTER_REQ`
2. AS 校验用户名唯一性
3. AS 落库 `salt + PBKDF2 hash`
4. AS -> C `REGISTER_REP`

## 7.2 登录取票

1. C -> AS `AS_REQ`
2. AS 校验用户名密码
3. AS 生成 `K_c_tgs`
4. AS 用 `K_tgs` 生成 TGT
5. AS 用 `K_user` 加密 `AS_REP_PART`
6. AS -> C `AS_REP`
7. C 用 `payload.userSalt + password` 派生 `K_user`
8. C 解 `AS_REP.payloadCipher`，拿到 `K_c_tgs`
9. C -> TGS `TGS_REQ`
10. TGS 校验 TGT + Authenticator
11. TGS 生成 `K_c_gs` 和 Service Ticket
12. TGS -> C `TGS_REP`
13. C 解出 `K_c_gs`
14. C -> GS `GS_AUTH`
15. GS 校验 Service Ticket + Authenticator
16. GS -> C `GS_AUTH_OK`
17. C 校验 AP_REP，认证完成

## 7.3 大厅与开局

1. C -> GS `ROOM_LIST_REQ`
2. C -> GS `ROOM_CREATE_REQ` 或 `ROOM_JOIN_REQ`
3. GS -> C `ROOM_CREATE_REP/ROOM_JOIN_REP`
4. GS -> 房内双方 `ROOM_STATE_PUSH`
5. C -> GS `ROOM_READY_REQ(isReady=true)`
6. 房主 -> GS `ROOM_START_REQ`
7. GS -> 双方 `MATCH_COUNTDOWN_PUSH`
8. GS -> 双方 `MATCH_EVENT_PUSH(MATCH_START)`

## 7.4 对战

1. C 周期性发 `PLAYER_INPUT_CMD`
2. GS 固定 Tick 消费输入
3. GS -> C `MATCH_SNAPSHOT_PUSH`
4. GS 按需要发 `MATCH_EVENT_PUSH`
5. 结束时 GS -> C `MATCH_RESULT_PUSH`

---

## 8. 异常、重连、登出、改密流程

## 8.1 断线重连

1. GS 检测到连接断开，标记会话 `DISCONNECTED_GRACE`
2. 在 `reconnectGraceMs` 内继续用空输入推进
3. Client 重新连 GS
4. C -> GS `RECONNECT_REQ(sessionId + ServiceTicket + 新 authenticator)`
5. GS 校验通过 -> `RECONNECT_REP`
6. GS -> `ROOM_STATE_PUSH`
7. GS -> `MATCH_SNAPSHOT_PUSH`

## 8.2 主动登出

1. C -> GS `LOGOUT_REQ`
2. GS `loginGen += 1`
3. GS 关闭当前 `ws_session`
4. GS -> C `LOGOUT_REP`
5. GS 断开连接

## 8.3 修改密码

1. C -> AS `CHANGE_PASSWORD_REQ`
2. AS 校验旧密码
3. AS 更新 `salt/hash`
4. AS `loginGen += 1`
5. AS -> C `CHANGE_PASSWORD_REP(needRelogin=true)`
6. 客户端清空 TGT / Service Ticket / session，重新 `AS_REQ`

---

## 9. 服务端校验顺序

## 9.1 AS 对 `REGISTER_REQ / AS_REQ / CHANGE_PASSWORD_REQ`

1. 解析 JSON
2. 检查 `ver / msgType / requestId / connSeq / timestampMs / clientId`
3. 校验 `connSeq` 是否按连接递增
4. 校验 `timestampMs` 是否在 ±30 秒内
5. 用 `keyId` 找 AS 私钥，解 `payloadCipher`
6. 校验明文字段完整性
7. 若 `(msgType, requestId, clientId)` 已命中过幂等缓存，直接回缓存结果
8. 执行注册 / 登录 / 改密逻辑
9. 生成响应并缓存

## 9.2 TGS 对 `TGS_REQ`

1. 外层公共字段检查
2. 检查 `ticket` 存在
3. 用 `K_tgs` 解 TGT
4. 检查 TGT `expireAtMs / renewUntilMs / loginGen / serviceName`
5. 用 `K_c_tgs` 解 `authenticator`
6. 检查 `clientId` 是否与 TGT 一致
7. 检查 `timestampMs` 是否在 ±30 秒内
8. 检查 `(ticketId, nonce, timestampMs)` 是否命中过 replay cache
9. 用 `K_c_tgs` 解 `payloadCipher`
10. 检查目标 `serviceName`
11. 生成 `K_c_gs`、Service Ticket、`TGS_REP`
12. 写 replay cache，回包

## 9.3 GS 对 `GS_AUTH`

1. 外层公共字段检查
2. 检查 `ticket` 存在
3. 用 `K_gs` 解 Service Ticket
4. 检查 `serviceName / expireAtMs / loginGen`
5. 用 `K_c_gs` 解 `authenticator`
6. 检查 `clientId` 是否与票据一致
7. 检查 `timestampMs` 是否在 ±30 秒内
8. 检查 replay cache
9. 创建 `sessionId`
10. 绑定 websocket -> session
11. 回 `GS_AUTH_OK`

## 9.4 GS 对业务消息

1. 外层公共字段检查
2. 校验 `sessionId` 已绑定
3. 校验 `loginGen` 仍有效
4. 校验 `mac`
5. 用 `K_c_gs` 解 `payloadCipher`
6. 检查消息类型对应的状态机合法性
   例如：

   * 没进房不能 `ROOM_READY_REQ`
   * 非房主不能 `ROOM_START_REQ`
   * 非 `PLAYING` 不能 `PLAYER_INPUT_CMD`
7. 对幂等消息检查 `(sessionId, msgType, requestId)` 缓存
8. 对 `PLAYER_INPUT_CMD` 检查输入序号 `seq`
9. 执行业务逻辑
10. 回 `REP` 或发 `PUSH`

Kerberos 对 replay cache 的要求很明确：如果服务丢失了允许时钟偏差窗口内的 replay cache，就必须在整个 skew 窗口内拒绝相关认证请求，否则会被票据重放攻击钻空子。你们这里至少要对 `TGS_REQ / GS_AUTH / RECONNECT_REQ` 做这层缓存。([RFC 编辑器][1])

## 12. 最后给你们的落地建议

这版协议最关键的几个点是：

1. **先把公共 Envelope 和加解密 helper 写死**
   Unity / Python 两边先统一：

   * `EncBlob_DES`
   * `DecBlob_DES`
   * `BuildMac`
   * `VerifyMac`

2. **先跑通 6 个最小消息**

   * `AS_REQ`
   * `AS_REP`
   * `TGS_REQ`
   * `TGS_REP`
   * `GS_AUTH`
   * `GS_AUTH_OK`

3. **再做房间**

   * `ROOM_CREATE_REQ/REP`
   * `ROOM_JOIN_REQ/REP`
   * `ROOM_READY_REQ/REP`
   * `ROOM_STATE_PUSH`
   * `ROOM_START_REQ/REP`

4. **最后把当前 `INPUT / SNAPSHOT` 升级成**

   * `PLAYER_INPUT_CMD`
   * `MATCH_SNAPSHOT_PUSH`
   * `MATCH_EVENT_PUSH`
   * `MATCH_RESULT_PUSH`

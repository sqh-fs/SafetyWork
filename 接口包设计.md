# SafetyWork 最终协议文档

## Slim Kerberos over WebSocket

## 1. 文档目的

本文档定义 SafetyWork 课程设计中 **AS、TGS、Client、Game Server** 之间的最终通信协议。

这是一版**瘦身后的协议**，遵循“less is more”原则，只保留真正影响功能实现的字段，避免为了“看起来完整”而引入过多顶层键。

这版协议的目标有三个：

1. 能完整跑通 Kerberos 认证流程
2. 能支持大厅、房间、准备、开始、对战、重连、登出
3. 结构足够简单，便于团队成员理解、调试和落地实现

---

## 2. 角色说明

系统中共有 4 个角色：

### 2.1 Client

玩家客户端，Unity 编写。
负责注册、登录、申请票据、连接 GS、发送输入、接收快照。

### 2.2 AS

认证服务器。
负责注册、登录、修改密码，并向客户端签发 **TGT**。

### 2.3 TGS

票据授予服务器。
负责验证 TGT，并向客户端签发访问指定 Game Server 的 **Service Ticket**。

### 2.4 GS

游戏服务器。
负责验证 Service Ticket、建立业务会话、管理房间、处理对战逻辑和断线重连。

---

## 3. 设计原则

### 3.1 顶层字段最少化

本协议只保留 8 个顶层字段，并且**不是每条消息都必须带满**。
不用的字段直接省略，不填空串，不填 `null`。

### 3.2 认证和业务分离

Kerberos 相关流程只解决“你是谁、你能不能连进来”。
房间、对战、快照等业务流程只在 GS 认证完成后进行。

### 3.3 尽量贴近现有代码

协议命名和消息结构尽量贴近你们当前仓库的实现风格，减少重构成本。

### 3.4 顶层不重复表达语义

不再保留这些字段：

* `ver`
* `service`
* `status`
* `requestId`
* `connSeq`
* `payloadCipher`
* `security`

原因很简单：
这些字段不是不能设计，而是对你们当前课设第一版没有必要。

---

## 4. 传输与编码约定

### 4.1 传输层

* 使用 **WebSocket**
* 使用 **UTF-8 文本帧**
* 每一帧都是一个完整 JSON 对象

### 4.2 顶层只允许 8 个键

```json
{
  "type": "TGS_REQ",
  "clientId": "cli-a-001",
  "sessionId": "sess-a-9001",
  "roomId": "A7K9Q2",
  "ticket": "Base64(...)",
  "auth": "Base64(...)",
  "payload": "....",
  "error": "...."
}
```

### 4.3 `payload` 的统一规则

`payload` 永远是一个**字符串**，只会有两种情况：

#### 情况 A：普通 JSON 字符串

用于大厅、房间、输入、快照、结果等普通业务消息。

例如：

```json
"payload": "{\"ready\":true}"
```

#### 情况 B：Base64 密文字符串

用于 AS/TGS/GS 认证阶段的敏感数据。

例如：

```json
"payload": "Base64(DES(...))"
```

### 4.4 时间单位

所有认证用时间戳都使用 **Unix 毫秒时间戳**。

### 4.5 房间号约定

本协议中 **`roomId` 直接就是房间码**。
不再区分 `roomId` 和 `roomCode`。

例如：

```json
"roomId": "A7K9Q2"
```

这意味着：

* 创建房间后，服务端直接生成一个 6 位房间号
* 加入房间时，客户端直接把这个值放进 `roomId`

---

## 5. 顶层字段说明

| 字段          | 含义                   | 是否必需         | 使用阶段                                         |
| ----------- | -------------------- | ------------ | -------------------------------------------- |
| `type`      | 消息类型                 | 是            | 所有消息                                         |
| `clientId`  | 客户端实例 ID             | 登录前必需，登录后可省略 | AS / TGS / GS 认证前                            |
| `sessionId` | GS 认证成功后的业务会话 ID     | GS 业务阶段必需    | 房间、对战、重连、登出                                  |
| `roomId`    | 房间号，同时也是入房码          | 房间/对战相关消息使用  | 房间、对战                                        |
| `ticket`    | TGT 或 Service Ticket | 认证相关消息使用     | AS_REP、TGS_REQ、TGS_REP、GS_AUTH、RECONNECT_REQ |
| `auth`      | Authenticator        | 认证相关消息使用     | TGS_REQ、GS_AUTH、RECONNECT_REQ                |
| `payload`   | 业务体字符串               | 大多数消息会使用     | 全阶段                                          |
| `error`     | 错误信息字符串              | 只在 ERROR 中使用 | 错误返回                                         |

## 6. 安全约定

## 6.1 用到的密钥

### `Kuser`

客户端根据用户密码派生出的密钥，用于解开 `AS_REP` 中返回的用户部分。

### `KcTgs`

客户端与 TGS 的会话密钥。
由 AS 下发，用于保护 `TGS_REQ / TGS_REP`。

### `KcGs`

客户端与 GS 的会话密钥。
由 TGS 下发，用于保护 `GS_AUTH / GS_AUTH_OK / RECONNECT_REQ / RECONNECT_REP`。

### `K_tgs`

AS 和 TGS 共享的长期密钥。
用于加密 TGT。

### `K_gs`

TGS 和 GS 共享的长期密钥。
用于加密 Service Ticket。

---

## 6.2 密文表示方式

为了减少顶层字段，不单独设置 `iv`。

统一约定：

### RSA 密文

```text
RSA(obj) = Base64( RSA_ENCRYPT( JSON(obj) ) )
```

### DES 密文

```text
DES(key, obj) = Base64( IV || DES_CBC( key, JSON(obj) ) )
```

也就是说：

* `ticket` 是 Base64 字符串
* `auth` 是 Base64 字符串
* 加密的 `payload` 也是 Base64 字符串

---

## 6.3 TGT 结构

TGT 由 AS 签发，客户端拿到后只能转发给 TGS，不能解开。

TGT 明文结构如下：

```json
{
  "userId": 10001,
  "clientId": "cli-a-001",
  "kcTgs": "Base64(8-byte-key)",
  "loginGen": 17,
  "exp": 1776679210100
}
```

字段含义：

* `userId`：用户 ID
* `clientId`：客户端实例 ID
* `kcTgs`：客户端和 TGS 的会话密钥
* `loginGen`：登录代数，用于让旧票据失效
* `exp`：过期时间

---

## 6.4 Service Ticket 结构

Service Ticket 由 TGS 签发，客户端拿到后转发给 GS。

明文结构如下：

```json
{
  "userId": 10001,
  "clientId": "cli-a-001",
  "service": "game/ws@10.0.0.12:9100",
  "kcGs": "Base64(8-byte-key)",
  "loginGen": 17,
  "exp": 1776657700100
}
```

字段含义：

* `userId`：用户 ID
* `clientId`：客户端实例 ID
* `service`：目标 GS 标识
* `kcGs`：客户端和 GS 的会话密钥
* `loginGen`：登录代数
* `exp`：过期时间

---

## 6.5 Authenticator 结构

### TGS_REQ / GS_AUTH 使用的认证器

```json
{
  "ts": 1776650510000,
  "nonce": "n2"
}
```

字段含义：

* `ts`：时间戳
* `nonce`：随机串

### RECONNECT_REQ 使用的认证器

```json
{
  "ts": 1776650535000,
  "nonce": "n3",
  "sessionId": "sess-a-9001"
}
```

多出的 `sessionId` 用于声明要恢复哪一个旧会话。

---

## 6.6 防重放规则

认证阶段统一使用 **30 秒时间窗口**：

* `abs(serverNow - auth.ts) <= 30000`
* 同一张 `ticket` 在 30 秒内，`(nonce, ts)` 不允许重复

---

## 6.7 登录代数 `loginGen`

每个用户在服务端维护一个 `loginGen`。

以下情况发生时，`loginGen += 1`：

* 成功登录
* 修改密码
* 主动登出
* 被顶号下线

作用：

* 旧 TGT 失效
* 旧 Service Ticket 失效
* 旧 GS 会话失效

# 8. 认证阶段协议

## 8.1 注册

### `REGISTER_REQ`

客户端发送给 AS：

```json
{
  "type": "REGISTER_REQ",
  "clientId": "cli-a-001",
  "payload": "Base64(RSA({\"username\":\"linhai\",\"password\":\"123456Aa!\"}))"
}
```

解密后的 `payload`：

```json
{
  "username": "linhai",
  "password": "123456Aa!"
}
```

### `REGISTER_REP`

```json
{
  "type": "REGISTER_REP",
  "payload": "{\"ok\":true,\"userId\":10001}"
}
```

---

## 8.2 登录并获取 TGT

### `AS_REQ`

```json
{
  "type": "AS_REQ",
  "clientId": "cli-a-001",
  "payload": "Base64(RSA({\"username\":\"linhai\",\"password\":\"123456Aa!\",\"nonce\":\"n1\"}))"
}
```

解密后的 `payload`：

```json
{
  "username": "linhai",
  "password": "123456Aa!",
  "nonce": "n1"
}
```

### `AS_REP`

```json
{
  "type": "AS_REP",
  "ticket": "Base64(TGT)",
  "payload": "{\"salt\":\"Base64(16B)\",\"iter\":100000,\"part\":\"Base64(DES(Kuser,{\\\"nonce\\\":\\\"n1\\\",\\\"kcTgs\\\":\\\"...\\\",\\\"exp\\\":1776679210100,\\\"loginGen\\\":17}))\"}"
}
```

其中：

* `ticket`：TGT
* `payload.salt`：派生 `Kuser` 用的盐
* `payload.iter`：PBKDF2 迭代次数
* `payload.part`：真正需要 `Kuser` 解开的部分

解开 `part` 后得到：

```json
{
  "nonce": "n1",
  "kcTgs": "Base64(8-byte-key)",
  "exp": 1776679210100,
  "loginGen": 17
}
```

---

## 8.3 修改密码

### `CHANGE_PASSWORD_REQ`

```json
{
  "type": "CHANGE_PASSWORD_REQ",
  "clientId": "cli-a-001",
  "payload": "Base64(RSA({\"username\":\"linhai\",\"oldPassword\":\"OldPass!23\",\"newPassword\":\"NewPass!23\"}))"
}
```

### `CHANGE_PASSWORD_REP`

```json
{
  "type": "CHANGE_PASSWORD_REP",
  "payload": "{\"ok\":true}"
}
```

服务端在此时必须：

* 更新密码
* `loginGen += 1`
* 使旧 TGT、旧 ST、旧 GS 会话全部失效

---

## 8.4 用 TGT 换取 Service Ticket

### `TGS_REQ`

```json
{
  "type": "TGS_REQ",
  "clientId": "cli-a-001",
  "ticket": "Base64(TGT)",
  "auth": "Base64(DES(KcTgs,{\"ts\":1776650500000,\"nonce\":\"n2\"}))",
  "payload": "Base64(DES(KcTgs,{\"service\":\"game/ws@10.0.0.12:9100\",\"nonce\":\"n3\"}))"
}
```

说明：

* `ticket`：客户端从 AS 拿到的 TGT
* `auth`：用 `KcTgs` 加密的认证器
* `payload`：用 `KcTgs` 加密的目标服务申请信息

### `TGS_REP`

```json
{
  "type": "TGS_REP",
  "ticket": "Base64(ServiceTicket)",
  "payload": "Base64(DES(KcTgs,{\"nonce\":\"n3\",\"kcGs\":\"Base64(8-byte-key)\",\"exp\":1776657700100}))"
}
```

解开 `payload` 后得到：

```json
{
  "nonce": "n3",
  "kcGs": "Base64(8-byte-key)",
  "exp": 1776657700100
}
```

---

## 8.5 用 Service Ticket 连接 GS

### `GS_AUTH`

```json
{
  "type": "GS_AUTH",
  "clientId": "cli-a-001",
  "ticket": "Base64(ServiceTicket)",
  "auth": "Base64(DES(KcGs,{\"ts\":1776650510000,\"nonce\":\"n4\"}))"
}
```

### `GS_AUTH_OK`

```json
{
  "type": "GS_AUTH_OK",
  "sessionId": "sess-a-9001",
  "payload": "Base64(DES(KcGs,{\"ts\":1776650510000,\"nonce\":\"n4\",\"exp\":1776657700100}))"
}
```

客户端需要检查：

* `payload.ts` 是否等于自己发出的 `auth.ts`
* `payload.nonce` 是否等于自己发出的 `auth.nonce`

只要能解密并且两者匹配，就说明 GS 确实持有 `KcGs`，完成双向认证。

---

## 8.6 断线重连

### `RECONNECT_REQ`

```json
{
  "type": "RECONNECT_REQ",
  "clientId": "cli-a-001",
  "sessionId": "sess-a-9001",
  "ticket": "Base64(ServiceTicket)",
  "auth": "Base64(DES(KcGs,{\"ts\":1776650535000,\"nonce\":\"n5\",\"sessionId\":\"sess-a-9001\"}))",
  "payload": "Base64(DES(KcGs,{\"lastProcessedSeq\":108}))"
}
```

### `RECONNECT_REP`

```json
{
  "type": "RECONNECT_REP",
  "sessionId": "sess-a-9001",
  "roomId": "A7K9Q2",
  "payload": "Base64(DES(KcGs,{\"ok\":true,\"phase\":\"PLAYING\",\"lastProcessedSeq\":108}))"
}
```

说明：

* 断线重连允许复用旧 `sessionId`
* 但必须重新生成新的 `auth`
* 服务端要检查旧会话是否仍在重连宽限期内

---

## 8.7 登出与强制下线

### `LOGOUT_REQ`

```json
{
  "type": "LOGOUT_REQ",
  "sessionId": "sess-a-9001"
}
```

### `LOGOUT_REP`

```json
{
  "type": "LOGOUT_REP",
  "sessionId": "sess-a-9001"
}
```

### `FORCE_LOGOUT`

```json
{
  "type": "FORCE_LOGOUT",
  "sessionId": "sess-a-9001",
  "payload": "{\"reason\":\"PASSWORD_CHANGED\"}"
}
```

可能的 `reason`：

* `PASSWORD_CHANGED`
* `LOGIN_REPLACED`
* `TICKET_EXPIRED`


### `ROOM_STATE.payload` 结构

```json
{
  "state": "WAITING",
  "ownerUserId": 10001,
```

其中：

房间成员列表，每个成员包含：

* `userId`
* `username`
* `ready`
* `online`

## 10.2 输入消息

### `INPUT`

```json
{
  "type": "INPUT",
  "sessionId": "sess-a-9001",
  "roomId": "A7K9Q2",
  "payload": "{\"seq\":101,\"moveX\":1.0,\"jumpPressed\":false,\"attackPressed\":true,\"downHeld\":false,\"dropPressed\":false,\"aimX\":0.92,\"aimY\":0.38,\"clientState\":\"Player_BasicAttackState\",\"clientGrounded\":true,\"clientJumpCount\":0,\"clientPosX\":12.53,\"clientVelX\":1.20}"
}
```

### `INPUT.payload` 字段说明

| 字段                | 说明           |
| ----------------- | ------------ |
| `seq`             | 输入序号，客户端单独递增 |
| `moveX`           | 水平移动输入       |
| `jumpPressed`     | 本帧是否按下跳跃     |
| `attackPressed`   | 本帧是否按下攻击     |
| `downHeld`        | 是否持续按住下方向    |
| `dropPressed`     | 是否触发下落/穿台    |
| `aimX`            | 瞄准方向 X       |
| `aimY`            | 瞄准方向 Y       |
| `clientState`     | 客户端当前状态名     |
| `clientGrounded`  | 客户端认为自己是否着地  |
| `clientJumpCount` | 客户端记录的跳跃次数   |
| `clientPosX`      | 客户端本地预测位置 X  |
| `clientVelX`      | 客户端本地预测速度 X  |

说明：

* `seq` 是对战输入序号，不是网络包序号
* 服务端应该以 `seq` 作为输入流排序和确认依据

---

## 10.3 快照消息

### `SNAPSHOT`

```json
{
  "type": "SNAPSHOT",
  "roomId": "A7K9Q2",
  "payload": "{\"tick\":240,\"lastProcessedSeq\":101,\"acceptedState\":\"Grounded\",\"acceptedGrounded\":true,\"acceptedJumpCount\":0,\"acceptedDrop\":false,\"serverPosX\":12.50,\"serverPosY\":3.00,\"serverVelX\":0.00,\"serverVelY\":0.00,\"enemyPosX\":17.20,\"enemyPosY\":5.10,\"enemyState\":\"BasicAttack\",\"rejectReason\":\"\"}"
}
```

### `SNAPSHOT.payload` 字段说明

| 字段                  | 说明            |
| ------------------- | ------------- |
| `tick`              | 当前服务器权威逻辑帧    |
| `lastProcessedSeq`  | 服务器已处理到的输入序号  |
| `acceptedState`     | 服务器接受的本地状态    |
| `acceptedGrounded`  | 服务器接受的着地状态    |
| `acceptedJumpCount` | 服务器接受的跳跃次数    |
| `acceptedDrop`      | 服务器接受的下落状态    |
| `serverPosX`        | 服务器权威位置 X     |
| `serverPosY`        | 服务器权威位置 Y     |
| `serverVelX`        | 服务器权威速度 X     |
| `serverVelY`        | 服务器权威速度 Y     |
| `enemyPosX`         | 对手位置 X        |
| `enemyPosY`         | 对手位置 Y        |
| `enemyState`        | 对手状态          |
| `rejectReason`      | 服务端拒绝输入或纠偏的原因 |

说明：

* `lastProcessedSeq` 是客户端回滚重算的关键字段
* `serverPosX/serverPosY` 是本地纠偏的关键字段
* `enemyPosX/enemyPosY/enemyState` 是对手显示的最小必要信息

## 10.4 结果消息

### `RESULT`

```json
{
  "type": "RESULT",
  "roomId": "A7K9Q2",
  "payload": "{\"winnerUserId\":10001,\"reason\":\"STOCK_ZERO\"}"
}
```

### `RESULT.payload` 字段

* `winnerUserId`：获胜者用户 ID
* `reason`：结束原因

建议 `reason` 取值：

* `STOCK_ZERO`
* `DISCONNECT_TIMEOUT`
* `SURRENDER`
* `FORCE_LOGOUT`

# 11. 正常流程时序

## 11.1 注册与登录

```text
Client -> AS   : REGISTER_REQ
AS -> Client   : REGISTER_REP

Client -> AS   : AS_REQ
AS -> Client   : AS_REP (返回 TGT + KcTgs)
```

## 11.2 获取 GS 票据

```text
Client -> TGS  : TGS_REQ (带 TGT + auth)
TGS -> Client  : TGS_REP (返回 Service Ticket + KcGs)
```

## 11.3 连接 GS

```text
Client -> GS   : GS_AUTH (带 Service Ticket + auth)
GS -> Client   : GS_AUTH_OK (返回 sessionId)
```

## 11.4 大厅与房间

```text
Client -> GS   : ROOM_LIST_REQ
Client -> GS   : ROOM_CREATE_REQ / ROOM_JOIN_REQ
Client -> GS   : ROOM_READY_REQ
Client -> GS   : ROOM_START_REQ
GS -> Clients  : ROOM_STATE / ROOM_START_REP
```

## 11.5 对战

```text
Client -> GS   : INPUT
GS -> Client   : SNAPSHOT
GS -> Clients  : RESULT
```

## 11.6 断线重连

```text
Client -> GS   : RECONNECT_REQ
GS -> Client   : RECONNECT_REP
GS -> Client   : SNAPSHOT
```

# 12. 实现约定

## 12.1 不用的字段不要发

例如 `ROOM_JOIN_REQ` 只需要：

```json
{
  "type": "ROOM_JOIN_REQ",
  "sessionId": "sess-b-9002",
  "roomId": "A7K9Q2"
}
```

不要额外再塞：

* `clientId`
* `ticket`
* `auth`
* `payload`
* `error`

---

## 12.2 统一先看 `type`

所有接收端解析流程都建议固定为：

1. 先解析顶层 JSON
2. 先看 `type`
3. 再看该消息需要哪些顶层键
4. 如果 `payload` 是普通 JSON 字符串，则再 `json.loads`
5. 如果 `payload` 是密文，则先 Base64 解码，再解密，再解析 JSON

---

## 12.3 客户端本地必须缓存的内容

### 登录后

客户端必须缓存：

* `clientId`
* `TGT`
* `KcTgs`

### 拿到 GS 票据后

客户端必须缓存：

* `ServiceTicket`
* `KcGs`

### GS 认证成功后

客户端必须缓存：

* `sessionId`

### 对战中

客户端建议缓存：

* 最近一次 `lastProcessedSeq`
* 最近一段未确认输入队列

这样才能支持断线重连和回滚同步。

---

## 12.4 房间状态更新策略

服务端只要遇到这些情况，就应该广播 `ROOM_STATE`：

* 创建房间成功
* 玩家加入房间
* 玩家离开房间
* 玩家准备/取消准备
* 玩家掉线/重连
* 房间开始游戏
* 对局结束回到房间

---

## 12.5 幂等策略

本协议不使用 `requestId`。
幂等由业务状态机保证：

* `ROOM_READY_REQ`：设置成 `true/false`，天然幂等
* `ROOM_JOIN_REQ`：若已经在该房间，可直接返回成功
* `ROOM_START_REQ`：若已开始，可直接回当前状态
* `LOGOUT_REQ`：若已登出，也可返回成功

---

# 13. 协议边界

本版协议**明确不包含**这些顶层字段：

```text
ver
service
status
requestId
connSeq
timestampMs
security
payloadCipher
mac
```

原因不是它们不专业，而是对你们当前这次课设第一版来说：

* 它们增加理解成本
* 它们增加实现成本
* 它们对核心功能帮助不大

如果后续想增强协议，可以在下一版再引入。

# 14. 最终结论

这版协议的核心思路可以总结成一句话：

> **认证阶段保留 Kerberos 核心语义，业务阶段使用最小 JSON 外层。**

也就是：

* 用 `ticket + auth` 完成认证
* 用 `sessionId` 管理 GS 会话
* 用 `roomId` 管理房间
* 用 `payload` 承载所有业务内容
* 用 `type` 作为唯一消息分发入口
